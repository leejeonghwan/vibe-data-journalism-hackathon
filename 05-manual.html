프로젝트 매뉴얼 — 운영 표준

한 사람의 손기술을 팀이 굴리는 시스템으로. 해커톤 운영부터 표준 작업·배포까지.

1~4강이 '어떻게 찾고 판단하는가'였다면, 5강은 '어떻게 반복 가능하게 만드는가'다.

데이터 기사 한두 개는 개인기로 만들 수 있다. 그러나 여러 명이 동시에, 같은 품질로, 꾸준히 만들려면 표준이 필요하다. 표준화된 워크플로우·저장소 템플릿·검증 절차가 그것이다. 이 매뉴얼은 그 표준을 정리한 것이다.

이 강의 구성
앞부분은 주말 2일 해커톤 운영(설계·팀·품질 게이트), 뒷부분은 표준 작업 절차(7단계 워크플로우·저장소 템플릿·코워크 활용·깃허브 배포)다. 취재 주제는 3강 공공데이터 100선에서 고른다.
5강 · 해커톤

주말 2일 부트캠프

금요일 저녁 사전 준비 + 토·일 이틀. 팀당 발행 가능한 대시보드 1개를 목표로.

전체 그림 — 100개는 누적으로

100개를 한 번에 만들 수는 없다. 분기 1회 부트캠프(회당 8~10팀 × 1~2개)와 사이사이 상시 트랙(숙련자가 개별 진행)으로 1~2년에 걸쳐 채운다. 부트캠프는 '교육+생산', 상시 트랙은 '생산'이다. 부트캠프 졸업생이 상시 트랙으로 올라가는 구조가 100개를 가능하게 한다.

월 5~10개
졸업생 상시 트랙
60~80개
1년 누적 목표

금요일 저녁 — 사전 준비 (2시간, 온라인 가능)

  • 코워크·깃허브 계정 세팅, 표준 템플릿 클론. 환경 문제는 여기서 모두 끝낸다 — 주말 본 행사에서 설치로 시간을 날리지 않게.
  • 100선에서 팀별 1지망·2지망 선택. 운영진이 데이터 접근성을 미리 확인해 배정한다.

1일차 (토) — 질문과 데이터

1

09:30 · 오리엔테이션 + 사례 해부 (60분)

대표 사례 하나(예: 콜린알포)를 열어 "질문 → 데이터 → 통제군 → 시각화 → 한계"를 거꾸로 분해한다. 좋은 데이터 기사의 골격을 머리에 새긴다.

2

10:30 · 질문 설계 워크숍 (90분)

"무엇을 증명할 것인가"를 한 문장으로. 검증 가능한 가설로 좁힌다. 여기서 막히면 나머지가 다 흔들린다 — 멘토가 가장 집중할 단계.

3

13:00 · 데이터 확보·정규화 (3시간)

100선이 가리키는 소스에서 원본 수집 → 코워크로 정규화(인코딩·결측·타입). 산출물은 깨끗한 data/processed.

4

16:30 · 1차 검증 + 데이 1 데모 (90분)

행수·합계·이상치 점검. 팀별 "오늘 확보한 데이터와 검증 결과" 3분 공유. 1일차 산출물 = 검증된 데이터셋 + 한 문장 가설.

2일차 (일) — 분석·배포·발표

5

09:30 · 분석·집계 (2.5시간)

가설을 수치로. 통제군 비교·시계열·상위 N·클러스터 등을 적용. data/*.json 생성.

6

13:00 · 대시보드 제작 (2.5시간)

표준 템플릿에 데이터 연결. 코워크에 "이 JSON으로 이런 차트"를 지시 → 단일 HTML 완성.

7

15:30 · 검증 게이트 + 배포 (90분)

발행 전 체크리스트(6강)를 통과해야 GitHub Pages 배포 허용. 통과 못 하면 '검증작' 등급으로 보류.

8

17:00 · 최종 발표 + 등급 부여 (60분)

팀별 5분: 질문·발견·한계. 심사가 실험/검증/발행 등급을 부여한다. 발행작은 슬로우뉴스 게재 예약.

설계 원칙
2일을 '교육'이 아니라 '가이드된 생산'으로 본다. 이론은 사례 해부 60분이면 충분하고, 나머지는 만들면서 배운다.

팀 구성과 멘토링

팀: 2~3명, 역할 분담

혼자 다 하면 초보는 무너진다. 작은 팀 안에서 역할을 나눈다 — 단, 고정이 아니라 서로 배우도록 순환한다.

  • 질문·검증 담당(기자 본령) — 무엇을 증명할지, 숫자가 말이 되는지. 가장 기자다운 역할.
  • 데이터 핸들러 — 수집·정규화·코워크 지시. 한 명이 손에 익히면 다음 프로젝트의 멘토가 된다.
  • 내러티브·비주얼 — 발견을 어떻게 보여 주고 쓸지. 제목·캡션·한계 서술.

멘토: 1인당 2~3팀

구간멘토가 개입할 곳
질문 설계가설이 검증 가능한지, 데이터로 답할 수 있는 크기인지 — 여기에 시간 최대 투입
데이터 정규화인코딩·조인 키·단위 함정(매출/주문/수주 혼동 등)
검증합계 교차검증, "원본에서 다시 세면 같은 수가 나오나"
해석상관을 인과로 오독하지 않기, 한계 명시
멘토 안티패턴
멘토가 대신 코드를 짜 주면 학습이 0이 된다. 멘토는 "코워크에게 어떻게 물을지"를 가르쳐야지, 답을 주면 안 된다.

5강 · 해외 노하우

교수법 노하우 — 해외 부트캠프에서

나이트센터·JournalismAI·NICAR가 비기술 기자를 가르칠 때 쓰는 검증된 방법.

1. 개념과 실습을 분리한다 — "말하기 vs 쓰기"

나이트센터(UT 오스틴)는 "AI를 말하는 것과 쓰는 것은 다르다"며 AI 교육을 개론 강좌와 실전 강좌로 나눴다. 우리도 개념 소개는 5분, 나머지는 직접 해보기로 못 박는다. 설명이 길어지면 손이 멈춘다.

2. 예제 데이터 말고 '본인 출입처 데이터'로

나이트센터 데이터 강좌는 매 실습을 수강생 자신의 취재 데이터로 진행하게 강제한다. 몰입도가 급상승한다. → 100선에서 각자 자기 출입처 데이터 하나를 골라 전 과정을 그 데이터로 끌고 가게 한다.

3. 최종 산출물 = '내 미니 AI 도우미'

JournalismAI Skills Lab(LSE)의 14주 과정은 이론이 아니라 참가자가 직접 프로토타입·도구를 만드는 것으로 끝난다. 게다가 'AI 기술'보다 사용자 중심 설계(프로덕트 싱킹)부터 시작한다. → 우리 과정도 마지막에 각자 출입처용 프롬프트 세트·워크플로를 결과물로 제출하게 한다.

4. 'AI 데이터 작업 4단계'로 고정 (NICAR)

NICAR 부트캠프는 스프레드시트를 ①정렬·수식 ②필터·추출 ③피벗(항목별 집계) ④표 결합 순으로 가르친다. 코워크에는 이 순서를 그대로 말로 시킨다.

코워크에 4단계로
1) "이 열을 큰 값부터 정렬해 줘"
2) "이 조건에 맞는 행만 추려 줘"
3) "항목별 합계를 표로 묶어 줘"      # 피벗
4) "이 두 표를 공통 키로 합쳐 줘"     # 결합(LOOKUP)
한 줄
코딩을 가르치는 게 아니라 "코워크에게 무엇을, 어떤 순서로 시킬지"를 가르친다. (출처: 나이트센터, JournalismAI, IRE/NICAR)
5강 · 표준 작업

표준 7단계 워크플로우

모든 프로젝트가 따르는 한 갈래. 초보도 이 순서만 밟으면 발행작이 나온다.

1

질문 정하기

"무엇을 증명할 것인가"를 검증 가능한 한 문장으로. 예: "신도시 발표 전, 지구 안 토지 거래가 통제군보다 비정상적으로 늘었는가?"

2

데이터 찾기

100선·공공데이터포털에서 1차 소스 확보. API·원본 파일 우선. 받자마자 manifest에 출처·해시 기록.

3

코워크로 정규화

인코딩(EUC-KR→UTF-8)·결측·타입·조인 키 정리. 산출물은 data/processed. 원본(raw)은 절대 덮어쓰지 않는다.

4

코워크로 분석·집계

전수 집계·통제군 비교·시계열·클러스터. build 스크립트가 data/*.json을 생성하게 만든다(다음에 다시 돌릴 수 있게).

5

검증

합계 교차검증, 원본 1건 이상 역추적, 이상치 확인. 결과를 _report에 남긴다. 6강 체크리스트 통과.

6

대시보드 + 깃허브 배포

표준 템플릿에 JSON 연결 → 단일 HTML → main push → GitHub Pages 공개.

7

수정·정정 루프

지적·새 데이터가 오면 data/*.json만 고쳐 다시 push. git diff에 수치 변경이 남아 정정 이력이 투명. 큰 변경은 PR로.

왜 이 순서인가
1·2·5가 기자의 일(질문·데이터·검증), 3·4·6이 코워크가 떠받치는 일이다. 초보는 기자의 일에 집중하고 기술은 AI가 맡는, 역할 분리가 이 7단계에 박혀 있다.

저장소 표준 템플릿

GitHub Template Repo로 만들어 'Use this template' 한 번에 복제.

data-journalism-template/
# 한 번 만들고 모두가 복제
├── index.html              # 단일 파일 대시보드 (data/ 로드)
├── README.md               # 아래 표준 양식
├── METHODOLOGY.md          # 질문·방법·통제군·한계
├── build.py                # raw → data/*.json + _report(검증)
├── data/
│   ├── manifest.json       # 원본 출처·해시·일시
│   ├── raw/                # 원본 (LFS 또는 Release)
│   ├── processed/          # 정규화 결과
│   ├── *.json              # 차트용 집계
│   └── _report.json        # 자동 검증 로그
├── LICENSE                 # 코드 MIT / 데이터 출처 귀속
└── .nojekyll

README 표준 양식 (필수 6항목)

  1. 한 줄 질문 — 이 프로젝트가 증명하려는 것
  2. 주요 발견 — 핵심 수치 3개 이내
  3. 데이터 출처 — 1차/2차, URL, 범위, 건수
  4. 방법 — 집계 방식·통제군 (상세는 METHODOLOGY.md)
  5. 재현 방법 — python build.py까지의 명령
  6. 한계 — 무엇을 입증하고 무엇은 못 하는가
메타데이터도 필수
저장소 생성 시 description·topics·license를 반드시 채운다. 검색·아카이브·발견성이 여기서 갈린다.

원본 데이터 보존 — '대용량이라 제외'를 버린다

원본을 빼면 시간이 지나 출처가 바뀔 때 재현·반박이 불가능해진다. 데이터 매니페스트에 각 원본의 URL·다운로드 일시·행수·SHA-256 해시를 기록한다. 용량은 0에 가깝지만 "이 분석이 어떤 데이터에 근거했는가"가 영구히 남는다. 원본 스냅샷은 Release 자산이나 Git LFS로 보존한다.

data/manifest.json
{
  "source": "한국언론진흥재단 정부광고 집행내역",
  "url": "https://www.kpf.or.kr/...",
  "files": [
    {"name":"2024.xlsx", "rows":201534,
     "sha256":"a1b2c3...", "fetched":"2026-05-22"}
  ]
}
5강 · 표준 작업

Cowork 활용 패턴

초보가 같은 품질의 지시를 내리게 하는 프롬프트 규율.

단계코워크에게 이렇게
정규화"이 CSV는 EUC-KR이고 N열에 결측이 있다. UTF-8로 바꾸고, 결측은 제거하지 말고 표시하고, 정규화 결과를 data/processed에 저장해라. raw는 건드리지 마라."
집계"(연도, 기관, 매체)로 그룹집계하고, 합계가 원본 총합과 일치하는지 검증한 뒤 차이를 보고해라."
통제군"지정 동 vs 비지정 동을 같은 기간으로 비교하고, 거시추세를 통제군으로 보정한 배율을 계산해라."
검증"무작위 5건을 원본에서 다시 찾아 집계값과 대조하고, 불일치가 있으면 멈추고 알려라."
대시보드"data/*.json만 읽는 단일 HTML로 만들어라. CDN은 Chart.js만. 백엔드·빌드 없이 브라우저에서 바로 열리게."

규율 3가지

  • "왜 그렇게 했는지 설명하게 하라." 코드만 받지 말고 근거를 묻는다 — 학습이 일어나는 지점.
  • 원본 불가침. raw 폴더는 읽기 전용. 모든 변형은 새 파일로.
  • 검증을 분석과 한 호흡으로. "집계하고 검증까지"를 한 지시에 묶는다.
스킬로 굳히기
위 지시문들을 코워크 스킬로 만들어 모든 팀 저장소에 넣으면, 초보가 긴 프롬프트를 외울 필요 없이 동일 품질이 보장된다.
5강 · 표준 작업

깃허브 몰라도 된다 — 코워크에게 '올려줘'

git 명령어를 외울 필요 없다. "올려줘"라고 말하면 코워크가 한다.

데이터 저널리즘의 마지막 두 단계(대시보드·배포)는 가장 기술적으로 보여 초보를 겁준다. 그런데 이것은 전부 말로 시킬 수 있다. 코딩도, git도 몰라도 된다. 코워크에게 자연어로 부탁하면 된다.

대시보드 — "만들어 줘"

코워크에게 (그대로 말하기)
"data 폴더의 json만 읽는 단일 HTML 대시보드를 만들어 줘.
 연도별 추이는 선그래프, 상위 기관은 막대그래프, 검색창도 넣어 줘.
 백엔드 없이 브라우저로 그냥 열리게. Chart.js만 CDN으로 써."
초보의 한 마디면 충분
"이거 보기 좋게 대시보드로 만들어 줘" → 결과를 보고 → "막대 말고 지도로" "색을 좀 차분하게" "모바일에서도 되게". 고쳐 가며 완성하면 된다. 차트 종류·코드는 코워크가 안다.

깃허브 — "올려 줘"

저장소 만들기, 커밋, 푸시, Pages 공개까지 — 말로 시킨다. 처음 한 번만 깃허브 계정과 로그인을 해 두면, 그다음부터는 "올려 줘"가 끝이다.

코워크에게 (그대로 말하기)
"이 폴더를 깃허브에 새 저장소로 올려 줘. 이름은 gov-ad-dashboard.
 그리고 GitHub Pages로 공개해서 주소 알려 줘."

수정도 같다. data/*.json 하나를 고친 뒤 "바뀐 거 다시 올려 줘"라고만 하면, 코워크가 변경분을 커밋·푸시하고 사이트가 자동 갱신된다.

왜 이 방식이 안전한가

걱정실제로는
"명령어 틀리면 다 날아가는 것 아닌가?"git은 버전관리 도구라 이전 상태가 다 남는다. 잘못 올려도 되돌릴 수 있다. "되돌려 줘"라고 하면 된다.
"공개하면 안 될 것을 올리면?"올리기 전 "올리면 안 될 파일(키·원본 개인정보)이 있는지 먼저 점검해 줘"라고 시킨다. .gitignore도 코워크가 만든다.
"비공개로 두고 싶은데""private 저장소로 올려 줘"라고 하면 나만 본다. 보도 후 "공개로 바꿔 줘".
딱 하나, 사람이 할 일
올리기 전 "공개해도 되는 데이터인가"의 판단만은 사람이 한다. API 키, 개인정보가 든 원본, 미검증 수치는 올리지 않는다. 기술이 아니라 판단의 문제이고, 그래서 기자의 몫이다.
