깃허브 원리 — 저장하고, 되돌리고, 공개한다

코드를 몰라도 된다. 깃허브가 무엇을 위한 도구인지 개념만 잡는다.

깃허브는 프로그래머만의 것이 아니다. 데이터 저널리즘에서는 작업을 안전하게 저장하고(버전), 남과 나누고(협업), 세상에 공개하는(배포) 도구다. 수주왕도, 신도시 토지 분석도 데이터·코드·대시보드를 깃허브로 공개했다. 이 강의 사이트 자체도 깃허브에 올라가 있다.

깃허브 = 구글 문서의 '버전 기록' + 무료 웹 호스팅

언제든 이전 상태로 되돌리고, 누가 무엇을 바꿨는지 남고, 완성물을 웹으로 공개할 수 있다. 기자에게 필요한 건 이 세 가지다.

이 강의 목표
깃허브를 '개발 도구'가 아니라 '취재 결과를 검증 가능하게 공개하는 곳'으로 이해한다. 세부 명령은 코워크에게 시키면 된다.
6강

세 가지 개념만 잡으면 된다

개념쉬운 비유하는 일
저장소 (Repository)프로젝트 폴더 하나한 취재의 데이터·코드·문서를 담는 통. 공개/비공개 선택.
커밋 (Commit)'저장' 스냅샷 + 메모변경을 시점별로 박제한다. "무엇을 왜 바꿨나"를 메시지로 남긴다.
푸시·풀 (Push/Pull)클라우드 동기화내 컴퓨터의 커밋을 깃허브로 올리고(push), 남의 변경을 받아온다(pull).
커밋은 '저장'이 아니라 '되돌릴 수 있는 저장'이다. 어제 잘 되던 상태로 언제든 돌아갈 수 있다는 것 — 이것이 버전 관리의 핵심이다.
6강

깃허브 페이지 = 무료 웹 배포

저장소에 index.html을 올리면, 그게 곧 웹사이트가 된다.

  • 공개 저장소에서 Settings → Pages를 켜면 아이디.github.io/저장소이름 주소로 사이트가 뜬다.
  • 서버·도메인·비용이 없다. 대시보드(단일 HTML)를 올리면 누구나 열어볼 수 있는 공개 대시보드가 된다.
  • 수주왕의 계약 검색, 신도시 토지 분석, 이 강의 사이트가 모두 이 방식이다.
이 사이트가 살아 있는 증거
지금 보고 있는 이 강의 페이지가 바로 깃허브 페이지로 배포된 결과물이다. 5강(시각화)에서 만든 대시보드를 여기에 얹으면 배포가 끝난다.
6강

코딩 없이 — 코워크로 올리기

기자는 "올려줘"라고 말하면 된다. 명령은 클로드가 짠다.

복붙 — 처음 올리기
이 폴더를 깃허브에 새 공개 저장소로 올려줘. 저장소 이름은 [프로젝트명]. 민감정보나 개인정보가 들어간 파일이 있으면 올리기 전에 먼저 알려줘. README도 간단히 만들어줘.
이런 결과 → 저장소가 생기고 파일이 올라간다. 클로드가 위험해 보이는 파일을 먼저 짚어준다.
복붙 — 수정 후 반영
방금 바꾼 내용을 깃허브에 반영해줘. 무엇을 왜 바꿨는지 커밋 메시지로 남기고, 푸시해줘.
깃허브 몰라도 된다
7강 매뉴얼의 "깃허브에 올려줘"와 같다. 개념(저장소·커밋·푸시)만 알면, 실행은 코워크가 대신한다.
6강

그래도 알아둘 명령 3개

터미널을 직접 쓸 때를 위해. 딱 이 셋이면 90%가 된다.

터미널git
# 1) 바뀐 파일을 담는다
git add -A
# 2) 스냅샷 + 메모로 박제
git commit -m "무엇을 왜 바꿨는지"
# 3) 깃허브로 올린다
git push
막히면
"권한 없음"·"잠금(index.lock)" 같은 에러가 나면 그 메시지를 그대로 코워크에 붙여넣고 물어본다. 대부분 한 줄로 해결된다.
6강

저널리즘에서 깃허브가 왜 중요한가

✅ 주는 것
  • 재현성 — 데이터·코드·방법을 공개하면 남이 검증할 수 있다.
  • 신뢰 — "어떻게 분석했는지"를 숨기지 않는다(BuzzFeed 감시기 사례).
  • 협업 — 여러 기자가 같은 프로젝트를 안전하게 함께 만진다.
  • 무료 배포 — 대시보드를 독자에게 바로 공개.
⛔ 아닌 것
  • 깃허브는 '개발자 전용'이 아니다.
  • 모든 걸 코딩으로 해야 하는 게 아니다(코워크).
  • 공개가 강제는 아니다 — 비공개 저장소도 된다.
6강

주의 — 공개의 무게

공개 저장소는 전 세계에 공개된다. 올리기 전에 반드시 확인한다.

  • 개인정보·취재원 — 실명·주민번호·연락처·미공개 제보 자료를 올리지 않는다.
  • 비밀번호·토큰 — API 키·비밀번호를 코드에 담지 않는다(한 번 올라가면 기록에 남는다).
  • 저작권·라이선스 — 원본 데이터의 재배포 조건을 확인한다.
  • 되돌려도 남는다 — 커밋 기록에 흔적이 남으니, 지웠다고 안심하지 않는다.
한 줄 원칙
"이 파일이 지금 인터넷 검색에 떠도 괜찮은가?"를 올리기 전에 스스로 묻는다.

실습 과제

  1. 연습용 폴더를 하나 만들고, 코워크로 공개 저장소에 올린다.
  2. 파일을 하나 고친 뒤, 커밋 메시지를 달아 다시 푸시한다.
  3. Settings → Pages를 켜고, 아이디.github.io/저장소가 열리는지 확인한다.
  4. 5강에서 만든 대시보드(단일 HTML)를 올려 공개 대시보드로 배포한다.
  5. 올리기 전에 "공개돼도 괜찮은가" 체크리스트를 스스로 통과시킨다.
