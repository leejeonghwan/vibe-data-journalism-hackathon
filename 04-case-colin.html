콜린알포 케이스 스터디

방대한 청구 데이터를 클로드 코워크와 코드로 다루며 '기사 가치'를 찾아간 실제 취재의 전 과정.

갈림길마다 무슨 판단을 했고, 그때 코워크에게 어떻게 시켰는지를 나란히 본다.

먼저 짚을 것
이 취재는 아직 진행 중이다. 여기 적힌 수치는 보도 확정본이 아니라 분석 과정의 작업 수치이고, 회사·기관에 대한 어떤 단정도 아니다. 이 강의 목적은 '결론'이 아니라 결론에 이르는 사고 과정과 기법을 전수하는 것이다.

이 케이스로 배우는 두 가지

데이터 저널리즘에는 가르치기 쉬운 절반과 어려운 절반이 있다. 이 케이스는 둘을 같은 화면에 나란히 둔다.

기사 가치 찾기 (판단)

방대한 데이터 앞에서 "무엇이 뉴스인가"를 가려내는 기자의 머리쓰기. 가설 설계, 킬러 팩트 포착, 한계 인정, 표현 수위 — 위임할 수 없는 사람의 일.

코워크·코드로 다루기 (실행)

27만 행을 실제로 받아내고 집계·시각화하는 기술. API 탐색, 다중 키, 페이지네이션, 대시보드 — 코워크에 위임할 수 있는 일.

각 갈림길은 상황 → 갈림길 → 판단 → 코워크에게 → 노하우 순으로 전개된다. 판단(파란 칸)과 실행(초록 칸)을 늘 짝으로 본다.

핵심 메시지
"코딩을 잘해야 데이터 저널리즘을 한다"는 것은 오해다. 코딩은 코워크가 한다. 기자가 잘해야 하는 것은 질문과 판단이다. 이 케이스가 그것을 보여 준다.

학습 지도 — 8개 갈림길

약 3일, 8단계. 그러나 단계는 매끄러운 직선이 아니라 매번 "이 길로 갈까 저 길로 갈까"의 연속이었다. 각 갈림길에서의 선택이 결국 기사를 만들었다.

갈림길 1

왜 콜린알포였나 — 주제 선택

세상의 모든 데이터를 분석할 수는 없다. 무엇을 고르느냐가 절반이다.

상황

콜린알포세레이트는 처방액이 큰데, 2020년 정부가 본인부담을 30%→80%로 올린 약이다. "정부가 효과를 의심한 약이 왜 아직도 많이 팔리나"라는 냄새가 있었다.

판단 — 좋은 주제의 조건

막연한 "이상한데"가 아니라 데이터로 검증 가능한 의심이어야 한다. ① 공개 데이터가 존재하고(건보 청구) ② 정부 조치라는 기준점이 있고(부담률 인상) ③ 가설을 숫자로 반증·입증할 수 있다. 세 조건이 맞아 "갈 수 있는 주제"로 판단했다.

코워크에게

아직 코드 단계는 아니다. 코워크는 여기서 "이 주제에 어떤 공개 데이터가 있고, 무엇을 검증할 수 있는지" 정찰에 쓴다. 데이터 존재 여부 확인이 주제 선택의 마지막 관문이다.

노하우
주제는 '흥미'가 아니라 '검증가능성'으로 고른다. 데이터가 없으면 아무리 좋은 의심도 기사가 안 된다. 초보에게는 이 판단을 카탈로그(데이터가 이미 확인된 주제 목록, 3강)가 대신해 준다.
갈림길 2

가설 먼저, 데이터 나중

데이터부터 들여다보면 길을 잃는다. 무엇을 증명할지 먼저 적는다.

상황

10년치 청구 데이터는 거대하다. 그냥 열면 "무엇을 봐야 하지"로 끝난다.

판단 — 가설 4개를 먼저

데이터를 보기 전에 검증할 가설을 못 박았다. ① 처방 급증 ② 청구 집중(소수 기관·지역) ③ 오리지널 과처방 ④ 지역 편중. 각 가설은 "참/거짓을 숫자로 가를 수 있는" 형태여야 한다. 가설이 있어야 어떤 집계를 할지가 정해진다.

코워크에게

가설을 design.md 분석 설계서로 문서화한다. 코워크에게 "이 4가설을 검증하려면 어떤 집계 차원(시간·지역·기관·상병)이 필요한지" 매핑시킨다. 설계서가 곧 작업 지시서가 된다.

코워크 지시 예
콜린알포 처방 데이터로 다음 4가설을 검증하려 한다. 각 가설마다 필요한 집계 단위·차원과 참/거짓을 가르는 판정 기준을 표로 정리해 달라. 가설: ①처방 급증 ②청구 집중 ③오리지널 과처방 ④지역 편중.
노하우
"데이터를 탐색하다 보면 뭔가 나오겠지"는 초보의 가장 흔한 함정이다. 가설→데이터가 원칙이다. 단, 가설은 도중에 수정될 수 있다(7번에서 실제로 더 센 것이 나온다). 가설은 출발선이지 결승선이 아니다.
갈림길 3

데이터 소스 뚫기 — 추측 말고 확인

API는 문서보다 실제가 다르다. 직접 만져 봐야 안다.

상황

심평원 OpenAPI에 12개 endpoint가 있다. 파라미터명을 문서만 보고 짐작했더니 응답이 비어 나왔다. 성분코드를 cmpnCd로 넣었는데 실제 필드는 gnlNmCd였다.

판단 — 추측을 신뢰하지 않는다

"문서에 이렇게 적혀 있으니 맞겠지"를 버린다. 빈 응답은 데이터가 없다는 뜻이 아니라 내가 잘못 묻고 있다는 뜻일 때가 많다. 직접 확인이 추측보다 항상 싸다.

코워크에게

Chrome 도구로 Swagger UI를 직접 열어 endpoint별 실제 파라미터·응답 모양을 확인한다. 코워크가 endpoint 레지스트리(endpoints.py)를 만들어 "어느 endpoint가 무엇을 반환하는지"를 코드로 고정한다.

발견: 파라미터명 불일치
# 문서/추측
params = {"cmpnCd": "138101ACS"}   # → 빈 응답
# Swagger에서 직접 확인한 실제 필드
params = {"gnlNmCd": "138101ACS"}   # → 정상 (콜린알포 일반명코드)
노하우
API 연동이 안 될 때 90%는 코드 버그가 아니라 파라미터·인증·페이지네이션의 오해다. "왜 안 되지"를 코워크와 함께 Swagger에서 한 줄씩 확인하면 대부분 30분 안에 풀린다.
갈림길 4

인증의 벽 — 막혔을 때 멈추지 않기

일일 호출 한도가 부족했다. 여기서 포기하면 취재는 끝난다.

상황

data.go.kr에서 키를 발급받았지만 일일 한도가 분석에 한참 모자랐다. 10년치를 받으려면 수만 콜이 필요한데 키 하나로는 며칠이 걸린다.

판단 — 제약을 우회 가능한 문제로 본다

"한도가 작으니 못 한다"가 아니라 "한도를 N배로 늘릴 방법"을 찾는다. 합법적 범위에서 가족·동료 명의로 키를 추가 발급해 풀(pool)로 묶었다. 막힘은 대개 설계로 푸는 문제다.

코워크에게

여러 키를 자동 순환하며 한도 소진 시 다음 키로 넘어가는 클라이언트(hira_client.py)를 코워크가 작성한다. "키 사이 콤마, 마침표·공백 없이"가 실수 포인트.

hira_client.py — 다중 키 자동 폴백 (개념)
KEYS = os.environ["HIRA_KEYS"].split(",")  # 콤마 구분, 공백X
def call(params):
    for key in KEYS:
        r = request(key, params)
        if not_exhausted(r): return r   # 한도 남은 키로 응답
    raise AllKeysExhausted   # 전부 소진 시 멈추고 알림
기자가 알아야 할 선
키 풀·크롤링은 해당 API의 이용약관과 합법 범위 안에서 한다. 공개 데이터라도 약관·개인정보·서버 부하의 선을 넘지 않는다. 막힘을 푸는 창의성과 규칙 준수는 함께 간다.
갈림길 5

작게 검증하고, 크게 수집한다

3만 콜을 무작정 돌리기 전에, 1콜로 확인한다.

상황

"30분이면 끝나겠지" 하고 시작했는데, 페이지네이션을 빠뜨려 실제론 2시간이 걸렸다. 거시 수집은 서울 25구 × 12 약효 × 120월 = 36,000콜, 정밀은 4,300콜에 27만 행 규모다.

판단 — 거시로 좁히고 정밀로 판다

처음부터 전부 정밀 수집하면 낭비다. 거시 시계열로 후보를 좁힌 뒤, 좁힌 곳만 정밀 드릴다운한다. "콜당 반환 행수가 큰 차원(상병)부터" 받는 것이 가성비가 좋다는 것도 작은 검증에서 배웠다.

코워크에게

① 1개월·1개구 smoke test로 응답·페이지네이션 확인 → ② 거시 크롤러 → ③ 정밀 크롤러. 그리고 매 콜마다 진척을 디스크에 저장(parquet)해, 끊겨도 이어받게 한다.

코워크 지시 예
본격 수집 전에 1개월·1개 지역만 호출해서 응답 구조와 페이지네이션 패턴을 확인해 달라. 그다음 전체 크롤러를 만들되 매 페이지 진척을 parquet으로 저장해서 중단돼도 마지막 지점부터 재개되게 해 달라.
가장 비싼 실수
진척 저장 없이 몇 시간짜리 수집을 돌리다 중간에 끊기는 것. 재개 가능한 크롤러는 선택이 아니라 필수다. 작업량 추정은 '콜 수'가 아니라 '콜 × 평균 페이지 수'로 한다.
갈림길 6

콜 안 쓰는 길 — OLAP 채널 발견

거시 통계를 한 푼의 콜도 안 쓰고 받을 수 있었다.

상황

거시 시계열에 3만 콜을 쓰고 있었는데, 심평원에 이미 집계된 통계를 내려주는 OLAP 별도 채널이 있었다.

판단 — "더 싼 길이 있지 않나"를 늘 의심

같은 데이터를 얻는 길은 보통 하나가 아니다. 원시 청구를 일일이 받아 직접 집계할 수도, 기관이 미리 집계해 둔 통계를 받을 수도 있다. 이미 집계된 것이 있으면 그것을 쓴다 — 거시는 OLAP, 정밀만 원시 수집으로 역할을 나눴다.

코워크에게

코워크에게 "이 포털에 집계 통계(OLAP/통계표) 채널이 따로 있는지 찾아보라"고 시켜 채널을 발견했다. 거시 시계열을 콜 소모 없이 확보하고, 아낀 한도를 정밀 드릴다운에 몰아줬다.

노하우
공공 포털은 보통 원시 데이터 API + 집계 통계(OLAP) + 파일 다운로드 세 갈래를 동시에 제공한다. 시작 전 "내가 원하는 것이 이미 집계돼 있나?"를 먼저 묻는다. 이 한 번의 질문이 며칠을 아낀다.
갈림길 7

숫자에서 헤드라인 — 어느 수가 기사인가

집계 결과는 수백 개. 그중 무엇이 1면인가.

상황

상병·지역·기관 세 차원으로 집계하니 숫자가 쏟아졌다. 2024년 청구 약 3,909억, 치매 핵심 상병 비중 12.0%, 1위 상병이 본태성 고혈압(572억)이고 알츠하이머는 2위(374억)…

판단 — 가장 반직관적인 수를 고른다

좋은 헤드라인은 "독자의 상식을 깨는 수"다. 치매약인데 1위 처방 상병이 고혈압, 그리고 사용된 상병 종류가 정부 인정 적응증의 358배(1,789개). "적응증 밖에서 광범위하게 쓰인다"는 한 문장이 이 모든 집계를 대표한다. 가설 ③(오리지널 과처방)보다 더 센 것이 여기서 나왔다.

코워크에게

A(상병)·B(시군구)·C(기관종) 세 트랙으로 분리 집계한다. 코워크에게 "적응증 목록과 실제 사용 상병을 대조해 적응증 외 비중·상병 종류 수를 계산"시켜 358배라는 수를 도출했다.

갈림길 8

데이터 밖에서 못 박기

청구 데이터만으로는 "회사의 위험 인식"을 못 잡는다. 밖에서 메운다.

상황

처방 데이터는 "얼마나 쓰였나"는 보여 주지만 "회사가 이 위험을 알고 있었나"는 못 보여 준다. 기사는 그 연결고리가 필요했다.

판단 — 데이터의 빈 곳을 외부 1차 자료로

회사가 환수 위험을 스스로 인정한 증거를 찾았다. 대웅바이오 사업보고서의 환수부채 1,035억(1년 매출의 80% 수준)을 직접 발견 — 회사가 위험을 회계로 인식했다는 결정적 정황이다. 옥시라세탐(퇴출된 유사 약)과 비교해 "같은 식약처가 왜 다른 결과를"이라는 정책 비판 각도 확보.

코워크에게

DART 사업보고서 주석에서 환수·충당부채 항목을 찾아 수치를 추출한다. 외신·학술(NICE·FDA 등)과 적응증 인정 여부를 대조한다. 데이터·공시·외신을 한 표로 합쳐 교차 근거를 만든다.

노하우
데이터 저널리즘은 '데이터만'의 저널리즘이 아니다. 청구 데이터(규모) + 기업 공시(인식) + 외신·학술(기준) + 인터뷰(반론)가 합쳐질 때 기사가 단단해진다. 데이터가 못 잡는 칸을 미리 알고 그 칸을 외부로 채운다.

여기가 초보에게 가장 위험한 구간이다. 숫자가 맞아도 해석이 틀리면 오보다.

표현 — 단정하지 말고 정황을 보여라

같은 사실도 어떻게 쓰느냐로 기사의 생사가 갈린다.

피할 표현 (단정)쓸 표현 (정황)
"효과가 없다""근거가 약함에도 광범위하게 사용된다"
"유착이 있다""구조적으로 의심할 정황이 있다"
"회사가 알고도 팔았다""회사는 환수부채를 회계에 인식했다"(사실 진술)

핵심은 두 가지다. 단정 대신 검증된 사실과 정황을 배치하고, 한계를 본문 안에 먼저 적어 반박을 사전 차단한다. 데이터의 한계를 숨기지 않고 드러내는 것이 오히려 신뢰를 만든다.

코워크의 또 다른 쓰임
코워크는 데스킹 보조로도 쓴다. "이 문장이 단정으로 읽히지 않는지, 회사가 반박할 여지가 어디인지 짚어 달라" — 표현 수위를 함께 점검한다. 단, 최종 책임과 판단은 사람(편집장)이 진다.
정리

노하우 12원칙

이 케이스에서 전이 가능한 것만 추렸다. 새 주제에도 그대로 쓰인다.

판단의 원칙 (기자의 일)

  1. 검증가능성으로 주제를 고른다. 데이터·기준점·반증가능성 세 조건.
  2. 가설을 먼저 적는다. 데이터부터 보면 길을 잃는다.
  3. 가장 반직관적인 수를 헤드라인 후보로. 단, 한계와 짝으로.
  4. 데이터의 빈 곳을 안다. 못 잡는 것(인식·동기)은 공시·외신·인터뷰로.
  5. 단정 대신 정황. 한계를 먼저 드러내 반박을 차단한다.
  6. 발견은 미리 못 적는다. 매뉴얼은 발견을 누락 없이 거치게 하는 안내서일 뿐이다.

실행의 원칙 (코워크의 일)

  1. 추측 말고 Swagger로 확인. 빈 응답은 보통 내 질문이 틀린 것.
  2. 작게 검증하고 크게 수집. 1콜 → 거시 → 정밀.
  3. 진척을 매번 저장. 재개 가능한 크롤러는 필수.
  4. 더 싼 길을 의심. 집계 통계(OLAP)가 이미 있을 수 있다.
  5. 인프라를 재사용 가능하게. 키워드만 바꿔 다음 약물에 곱하기.
  6. AI 수치는 사람이 역추적 검증. 1건 이상 원본 대조 후 보도.
한 문장으로
기자는 1~6(판단)에 집중하고, 7~12(실행)는 코워크가 떠받친다. 코딩 실력이 아니라 이 역할 분리를 아는 것이 데이터 저널리스트의 출발점이다.
정리

이제 직접 — 같은 틀을 새 약물·새 영역에

콜린알포는 끝이 아니라 첫 케이스다. 같은 골격이 다음을 만든다.

이 취재의 인프라(크롤러·분석 모듈·대시보드)는 키워드만 바꾸면 다른 약물·다른 분야로 즉시 재사용된다. 8종 묶음 크롤러가 30분 만에 만들어진 이유다. 첫 프로젝트는 "새 길을 뚫는 것"이 아니라 "이미 뚫린 길을 한 번 걸어 보는 것"이다.

해커톤/강의 첫 과제로 권하는 흐름

  1. 3강 100선에서 데이터가 이미 확인된 주제 하나를 고른다(예: 다른 의심 약물, 비급여 항목).
  2. 콜린알포의 4가설 틀을 그대로 빌려 내 주제에 맞게 4개를 적는다.
  3. smoke test → 거시 → 정밀 순서로, 재사용 크롤러를 키워드만 바꿔 돌린다.
  4. 가장 반직관적인 수를 찾고, 5가지 cross-check를 통과시킨다.
  5. 표준 템플릿 대시보드에 연결해 GitHub Pages로 올린다(5강).
멘토에게
초보가 1~6(판단)에서 막히는 것은 정상이다 — 거기가 기자의 일이고, 멘토가 붙어야 할 곳이다. 7~12(실행)에서 막히면 코워크에게 더 잘 물어보게 도와라. 멘토가 대신 코드를 짜 주면 학습은 0이 된다.
